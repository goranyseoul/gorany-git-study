# 9. 통합 테스트

> 전체 시스템 **통합 테스트** 단계
> 실패 시 6번부터 반복

```

                       ▼
```

---

## 입력
- 배포된 앱 또는 로컬 서버
- `docs/1-요구사항/` (사용자 시나리오)
- `docs/7-프론트엔드/user-scenario.md`
- `docs/8-백엔드/api-spec.md`

---

## Phase 1: E2E 테스트 (@9a-integration-tester)

1. 프론트엔드 + 백엔드 통합 환경 구성
2. E2E 테스트 작성 (Playwright)
3. 사용자 시나리오별 검증
4. API 통합 테스트

**산출물:** `e2e/`, `docs/9-테스팅/test-results.md`

---

## Phase 2: QA 분석 (@9b-qa-analyst)

1. 테스트 결과 분석
2. 버그 분류 (Critical / Major / Minor)
3. QA 리포트 작성
4. **반복 필요 여부 결정**

**산출물:** `docs/9-테스팅/qa-report.md`

---

## 피드백 루프

테스트 실패 시 `qa-report.md`에 다음 정보 포함:

```markdown
## 반복 필요 여부
- [x] 6번(개발 기획)부터 재시작 필요
- [ ] 7번(프론트엔드)만 수정
- [ ] 8번(백엔드)만 수정

## 수정 필요 항목
| 영역 | 버그 ID | 설명 | 우선순위 |
|------|--------|------|---------|
| BE | BUG-001 | 로그인 API 500 에러 | Critical |
| FE | BUG-002 | 버튼 클릭 안됨 | Major |
```

---

## 반복 조건

| 상황 | 액션 |
|------|------|
| **Critical 버그** | 6번부터 재시작 |
| **Major 버그** | 7/8번 해당 영역만 수정 |
| **Minor 버그** | 9번에서 직접 수정 |
| **버그 없음** | ✅ 완료 → 배포 |

---

## 출력
- `e2e/` - E2E 테스트 파일
- `docs/9-테스팅/test-results.md` - 테스트 결과
- `docs/9-테스팅/qa-report.md` - QA 리포트 (반복 여부 포함)
