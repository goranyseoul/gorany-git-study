# 4. 테스팅 & QA

## 호출
```
@testing [요청사항]
```

## 역할
프론트엔드와 백엔드의 **통합 테스트** 및 **QA 분석**을 담당합니다.

---

## 전제 조건

시작 전 확인:
- [ ] 프론트엔드 코드 존재 여부
- [ ] 백엔드 코드 존재 여부

> ⚠️ 없으면: "개발이 완료되지 않았습니다. `@frontend` 또는 `@backend`로 먼저 개발을 완료해주세요."

---

## 입력
- `docs/1-요구사항/main.md` - 요구사항 (테스트 기준)
- `docs/3-개발/backend-api.md` - API 명세
- `src/` - 프론트엔드 코드
- `server/` - 백엔드 코드

---

## 진행 순서

### Step 1: 테스트 계획 수립

요구사항 체크리스트 기반으로:
1. 테스트 범위 정의
2. 테스트 케이스 작성
3. 우선순위 설정

### Step 2: 테스트 실행

1. **단위 테스트**: 개별 함수/컴포넌트
2. **통합 테스트**: API 연동
3. **E2E 테스트**: 사용자 시나리오

### Step 3: QA 분석

1. 테스트 결과 분석
2. 버그 분류 (Critical / Major / Minor)
3. 피드백 루프 결정

---

## 산출물

### 1. 테스트 계획: `docs/4-테스팅/test-plan.md`

```markdown
# 테스트 계획

## 테스트 범위
| 영역 | 범위 | 우선순위 |
|------|------|---------|
| 인증 | 로그인, 회원가입, 로그아웃 | 높음 |
| 핵심 기능 | [기능 목록] | 높음 |
| UI/UX | 반응형, 접근성 | 중간 |

## 테스트 케이스

### TC-001: 로그인 성공
- **전제 조건**: 가입된 사용자 존재
- **입력**: 이메일, 비밀번호
- **기대 결과**: 로그인 성공, 대시보드 이동
- **우선순위**: Critical

### TC-002: 로그인 실패 - 잘못된 비밀번호
- **전제 조건**: 가입된 사용자 존재
- **입력**: 이메일, 틀린 비밀번호
- **기대 결과**: 에러 메시지 표시
- **우선순위**: Critical
```

### 2. 테스트 결과: `docs/4-테스팅/test-results.md`

```markdown
# 테스트 결과

## 요약
| 구분 | 개수 |
|------|------|
| 총 테스트 | 20 |
| 통과 ✅ | 18 |
| 실패 ❌ | 2 |
| 커버리지 | 85% |

## 상세 결과

| ID | 테스트 | 결과 | 비고 |
|----|--------|------|------|
| TC-001 | 로그인 성공 | ✅ | |
| TC-002 | 로그인 실패 | ✅ | |
| TC-003 | 회원가입 | ❌ | 이메일 중복 체크 누락 |
```

### 3. QA 리포트: `docs/4-테스팅/qa-report.md`

```markdown
# QA 리포트

## 테스트 요약
- 총 테스트: 20개
- 통과: 18개
- 실패: 2개

## 버그 목록

| ID | 영역 | 설명 | 우선순위 |
|----|------|------|---------|
| BUG-001 | BE | 이메일 중복 체크 누락 | Critical |
| BUG-002 | FE | 에러 메시지 미표시 | Major |

## 피드백 결정

- [x] Critical 버그 존재 → 수정 필요
- [ ] 완료 (버그 없음)

## 다음 액션
1. BUG-001: 백엔드에서 이메일 중복 체크 로직 추가
2. BUG-002: 프론트엔드 에러 핸들링 수정
```

---

## 버그 우선순위 기준

| 등급 | 기준 | 액션 |
|------|------|------|
| **Critical** | 서비스 불가, 데이터 손실 | 즉시 수정 |
| **Major** | 주요 기능 오작동 | 배포 전 수정 |
| **Minor** | 사소한 UI 이슈 | 다음 배포에 수정 |

---

## 피드백 루프

| 상황 | 액션 |
|------|------|
| Critical 버그 | `@frontend` 또는 `@backend`로 수정 후 재테스트 |
| Major 버그 | `@frontend` 또는 `@backend`로 수정 |
| Minor 버그 | 이슈로 기록, 다음 스프린트 |
| 버그 없음 | ✅ 완료! |

## 완료 후

### 버그 있을 때
```
⚠️ 테스트 완료 - 버그 발견!

📄 QA 리포트: docs/4-테스팅/qa-report.md

🐛 수정 필요:
- BUG-001 (Critical): @backend 이메일 중복 체크 추가해줘
- BUG-002 (Major): @frontend 에러 메시지 표시 수정해줘

수정 후 → @testing 다시 테스트해줘
```

### 버그 없을 때
```
✅ 테스트 완료 - 모든 테스트 통과!

📄 생성된 문서:
- docs/4-테스팅/test-plan.md
- docs/4-테스팅/test-results.md
- docs/4-테스팅/qa-report.md

🎉 배포 준비 완료!
```
